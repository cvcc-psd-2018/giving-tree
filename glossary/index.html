    Glossary

    This is a collection of commonly-used terms in programming. If you're reading a resource, such as a tutorial on this site, and a word makes you go, "huh?", try looking for it here.

    Tip: Use Ctrl+F to search for a term. They're broken up into chapters, so you can also search for a chapter.
    Don't see a term you want defined? Create a new issue here with the tag "glossary term".

    Java Chapter 1

    This chapter mostly covered basic computing terms not exclusive to programming. If you're into gaming, there's a good chance you've already heard most of these.

    Input

    An input is a source of data that's fed into a program. It can also be used in a more general sense to refer to the data itself.

    Output

    Output is data that a program publishes, whether directly to a user or to another program or component for further processing. It can also be used to describe a component that's specifically made to handle that data.

    Central Processing Unit (CPU)

    The CPU is the part of a computer that runs active calculations for most computing tasks. It can also be used to describe the CPU's processing power.

    Random Access Memory (RAM, Primary Memory)

    The RAM is the part of a computer that holds data that's fed directly to and from the CPU. Most programs, along with all their variables, are kept in RAM while they're running.

    Archive (Secondary Storage)

    Archive memory is where most files are kept, as well as other essentials like the operating system. It uses very little to no electricity to keep things stored in archive, so this is where we put things that we want to be able to keep when the computer is turned off. Your HDD (hard disk drive, or hard drive), SSD (solid-state drive), SD cards, USB drives, floppy disks, CDs, casettes, and so on are all things used to store archive memory.

    Algorithm

    An algorithm is a generalized mathematical formula that can be applied to solve a specific problem. Technically speaking, running a program is one massive algorithm, but we typically mean it in a more focused way. They can be incredibly simple (such as "the product of a number and two is the sum of the number plus itself" [or x times 2 = x + x]), or so impossibly complex that no human is physically capable of totally understanding the underlying mathematics (such as the YouTube recommendation system, and most other forms of artificial intelligence).

    Compiler

    A compiler transforms code written in a programming language (such as C) into a form that can be directly understood by the CPU. It is itself a program.

    Boolean

    A boolean is a "binary" value, meaning it only has two possible values -- True or False, yes or no, one or zero. In most programming languages with boolean support such as Java, it is a primitive data type that can be either True or False.

    Relational Operator

    A relational operator is an operator that makes a comparison between its two operands, and returns a boolean True or False. They're mostly mathematical comparisons, such as equals (==), checking if the numbers are identical, or not equals (!=), checking if the numbers are different, greater than, greater than or equal to, less than, and less than or equal to (> < >= <=, respectively), which should all be more-or-less self-explanatory if you've taken a math class in the past five years.

    Logical Operator

    A logical operator processes a number of boolean values, and outputs a new boolean based on them. The three major logical operators are AND (&&), OR (||), and NOT (!). AND only returns TRUE if both its operands are TRUE, while OR will return TRUE if either operand is true. NOT only takes one operand, and returns the opposite boolean value (meaning that TRUE becomes FALSE and vice versa).

    Short-Circuit Evaluation

    Short-circuiting is when a logical operator only evaluates one operand. Since AND only needs one operand to be false for the whole expression to be false, it immediately evaluates to false without even checking the other operand if the left operand is false. OR, on the other hand, only requires one statement to be true for the whole statement to evaluate as such, so it also short-circuits if the left operand is true, outputting true without even checking the right operand.

    This is all a little bit confusing, so it can be boiled down to this in practice: The left operand in an AND statement should be the one most likely to be false, and in an OR statement, should be the one most likely to be true.

    Selection Structures

    Selection structures are the most basic control flow structures, which change how the flow of logic in a program works. These in particular come in the form of if/else and switch statements.

    IF statements come in the form of if (condition) {statements}.
    The condition, in most languages, should be a boolean. If the boolean evaluates to TRUE, then the statements run. Otherwise, they don't. It's a lot like English in that regard -- "if we have coffee, get me some" is much the same as "if (coffee > 0) {getCoffee();}".
    ELSE statements only run if the condition of the IF statement is false -- in other words, only if the IF statement doesn't run. It's almost like making a second IF statement with !condition as its own condition.
    These can be paired together with an ELSE IF statement, which only has its condition checked if the previous IF statement was false. You can go to extreme depths with this, with a huge chain of IF/ELSE IF/[...]/ELSE IF/ELSE.
    Fun fact: Most software on the market that touts "Advanced Artificial Intelligence" just has a bunch of really long IF/ELSE chains. If you have the patience and the inclination, you too can make software that looks like it's really thinking. Try it out sometime. It's incredibly tedious, but it's possible!

    Switch statements are quite similar to IF/ELSE statements. Given a single condition of almost any primitive data type, you can make a series of CASEs -- each of which are similar in function to "if (condition == thisCase)". If the condition matches a case, then the code starts running there, but doesn't stop until it reaches a BREAK statement. This means that, if you write multiple CASEs without a BREAK in between them, then the code won't skip over the cases that don't match your condition. This might actually be a good thing sometimes, which is why it happens, but it can cause problems just as often as it helps solve them if you aren't careful.
    SWITCHes can also have a DEFAULT case, which the code jumps to when none of the other CASEs match.

    Ternary Operator

    The ternary operator looks and sounds complicated, but it's actually really simple! It has the syntax condition ? ifTrueDoThis : ifFalseDoThis. It's identical to an IF/ELSE statement of if (condition) {ifTrueDoThis} else {ifFalseDoThis}. You don't strictly need to use or even necessarily know it, but it can be a handy way to save time and space if you have a small conditional statement to make.
